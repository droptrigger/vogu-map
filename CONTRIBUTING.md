# Соглашения по написанию кода

## Коммиты

Проект следует спецификации [Conventional Commits](https://www.conventionalcommits.org/).

Формат сообщения коммита:

```
<тип>(<область>): <описание>
```

- **тип** — обязателен
- **область** — опционален, указывает на область изменений
- **описание** — с маленькой буквы, без точки в конце

### Типы

| Тип | Назначение |
|-----|------------|
| `feat` | Новая функциональность |
| `fix` | Исправление ошибки |
| `docs` | Изменения в документации |
| `style` | Форматирование (пробелы, отступы, точки с запятой) без изменения логики |
| `refactor` | Рефакторинг кода (не фича и не фикс) |
| `perf` | Оптимизация производительности |
| `test` | Добавление или изменение тестов |
| `build` | Сборка, зависимости, конфигурация пакетов |
| `ci` | Изменения в CI/CD пайплайнах |
| `chore` | Технические задачи, не влияющие на продакшен |
| `revert` | Откат предыдущего коммита |

### Примеры

```
feat(building): добавление контроллера работы с корпусами
feat(room): добавление новой таблицы rooms
docs: добавление MIT-лицензии
chore: обновление зависимостей до актуальных версий
```

## Ветки

Формат имени ветки:

```
<тип>-<номер задачи>/<область>
```

Примеры:
```
feat/room
fix-102/building
docs/readme
refactor-88/room
```

---

## Merge Request

- Один MR — одна задача или связанная группа задач
- Заголовок MR повторяет формат коммита: `feat(room): добавление новой таблицы Rooms`
- Ветка вливается в `main` через MR, прямые пуши в `main` запрещены
